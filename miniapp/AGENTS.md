# silver-wisdom 小程序前端提示词

## 1. 适用范围
- 本文件仅用于 `miniapp/` 小程序端相关任务。
- 仅在小程序目录内生成页面、组件、状态、接口调用、授权流程与端能力适配代码。
- 不要把管理后台的高信息密度表格交互模式直接搬到小程序页面。

## 2. 小程序设计目标
- 优先关注移动端单手操作、首屏加载速度、路径简洁、弱网容错、授权与登录体验。
- 输出内容优先覆盖：页面流转、用户触达链路、组件布局、接口调用、状态反馈、端能力限制。
- 新页面优先考虑列表、详情、表单、确认操作、分享、下拉刷新、触底加载等移动端常见场景。

## 3. 代码组织建议
- 优先按 `pages`、`components`、`services/api`、`utils`、`store` 或项目现有等价结构组织代码。
- 页面负责业务流程与交互编排；可复用业务块沉淀到组件；请求层统一管理接口调用与错误处理。
- 避免把复杂业务和大量视图逻辑都堆在单个页面文件中。

## 4. 小程序输出规范
- 方案说明需包含：页面归属、用户操作路径、接口依赖、授权点、异常场景。
- 页面生成时优先说明：导航入口、首屏信息、按钮行为、空态、加载态、弱网提示、登录态处理。
- 调接口时必须说明请求参数、响应字段映射、页面刷新时机和失败反馈方式。
- 涉及登录态或鉴权时，前端只描述 token 使用、登录跳转、未登录拦截与重试体验，不替代后端鉴权。

## 5. 与后端协作边界
- 若缺少接口，先给出适合移动端的接口契约建议，字段命名尽量简洁明确。
- 不主动输出后端 Service、Mapper、数据库实现，除非用户明确要求一并设计。
- 对需要后端支持的能力，要明确标注依赖项，例如登录态校验、内容分页、课程状态查询等。

## 6. 交付要求
- 代码示例需给出可落地的页面名、组件名、方法名、字段名。
- 优先保证交互清晰和移动端可用性，不照搬 PC 管理后台的布局与组件思路。
- 如需求与本提示词冲突，以用户当次明确指令为最高优先级。
