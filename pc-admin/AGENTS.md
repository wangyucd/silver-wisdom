# silver-wisdom 管理后台前端提示词

## 1. 适用范围
- 本文件仅用于 `pc-admin/` 管理后台前端相关任务。
- 仅在管理后台目录内生成页面、组件、路由、状态管理、接口调用与样式代码。
- 不要把后端分层、Mapper、事务、Redis、Nacos、Sa-Token 实现细节直接写进前端代码。

## 2. 目标导向
- 管理后台优先关注：信息密度、表格效率、筛选检索、状态操作、权限控制、可维护性。
- 输出内容优先覆盖：页面结构、组件拆分、路由设计、接口调用封装、状态流转、异常提示。
- 新页面应优先考虑新增、编辑、详情、列表、筛选、分页、批量操作等后台常见场景。

## 3. 代码组织建议
- 优先按 `pages`、`components`、`api`、`types`、`hooks` 或项目现有等价结构组织代码。
- 页面负责装配布局与交互流程；复用性 UI 片段沉淀到 `components`；接口请求统一收敛到 `api` 层。
- 表单、表格、弹窗、抽屉等高复用模块优先做组件化封装，避免页面内堆积过长逻辑。

## 4. 管理后台输出规范
- 方案说明需包含：页面归属、用户操作路径、接口依赖、状态变化、边界场景。
- 页面生成时优先说明：筛选项、表格列、操作按钮、空态、加载态、异常态、权限控制点。
- 调接口时必须说明请求参数、响应字段映射、列表刷新时机和失败提示方式。
- 涉及权限时，前端只做展示与操作级收敛，不替代后端真实鉴权。

## 5. 与后端协作边界
- 若缺少接口，先给出接口契约建议：URL、Method、入参、出参、关键字段含义。
- 若用户任务聚焦前端联调，不主动扩写后端实现代码；只在必要时补充接口约定。
- 不根据前端需要擅自假设后端已有复杂能力；需明确“依赖后端提供”的字段和行为。

## 6. 交付要求
- 代码示例需给出可落地的组件名、函数名、类型名、字段名。
- 优先保证后台场景的可读性与维护性，不为了炫技引入复杂视觉表现。
- 如需求与本提示词冲突，以用户当次明确指令为最高优先级。
